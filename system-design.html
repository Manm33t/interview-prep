Distributed Systems Fundamentals

CAP Theorem — What It Really Means

CAP: a distributed system can guarantee at most two of three: Consistency, Availability, Partition Tolerance. Critical nuance: partition tolerance is not optional — network partitions happen in any real distributed system. The real choice: during a partition, sacrifice C or A?

CP Systems — Consistent & Partition Tolerant

During partition, refuse to respond rather than return stale data. Examples: ZooKeeper, HBase, etcd, Redis (w/ WAIT), MongoDB (majority write concern). Use when: financial transactions, inventory, leader election.

AP Systems — Available & Partition Tolerant

During partition, keep serving with potentially stale data. Examples: Cassandra, DynamoDB (default), CouchDB, DNS. Use when: shopping carts, social feeds, metrics — serving stale data > serving no data.

PACELC + Consistency Models

Linearizability

Strongest: after write completes, all reads see it. ZooKeeper, etcd. High latency.

Causal Consistency

Causally related ops seen in order by all. Concurrent ops may differ. MongoDB causal sessions.

Eventual Consistency

No new writes → replicas converge. No timing guarantee. DNS, Cassandra default.

Scalability Patterns

Vertical Scaling (Scale Up)

Add more CPU/RAM to existing machine. Simpler, no distributed coordination. Hard ceiling + single point of failure. Use for: databases, monoliths in early growth.

Horizontal Scaling (Scale Out)

Add more machines. Requires stateless app tier (session in Redis). Theoretically unbounded. Needs load balancing, service discovery. Use for: API servers, stateless microservices, Lambda workers.

Consistent Hashing

Classic key % N requires remapping all keys when N changes. Consistent hashing maps keys AND nodes onto a hash ring. Adding/removing a node remaps only K/N keys on average. Virtual nodes (150 per physical node) ensure even distribution — without them, non-uniform arc sizes cause 2-5× traffic skew.

class ConsistentHashRing {
          constructor(virtualNodes = 150) {
            this.ring = new Map();   // hash → node
            this.sortedKeys = [];
            this.virtualNodes = virtualNodes;
          }
          addNode(node) {
            for (let i = 0; i < this.virtualNodes; i++) {
              const hash = this._hash(`${node}#${i}`);
              this.ring.set(hash, node);
              this.sortedKeys.push(hash);
            }
            this.sortedKeys.sort((a, b) => a - b);
          }
          getNode(key) {
            const hash = this._hash(key);
            let lo = 0, hi = this.sortedKeys.length - 1;
            while (lo < hi) {
              const mid = (lo + hi) >> 1;
              if (this.sortedKeys[mid] < hash) lo = mid + 1;
              else hi = mid;
            }
            const nodeHash = this.sortedKeys[lo] >= hash ? this.sortedKeys[lo] : this.sortedKeys[0];
            return this.ring.get(nodeHash);
          }
          _hash(str) {
            let hash = 0;
            for (const c of str) hash = ((hash << 5) - hash) + c.charCodeAt(0);
            return hash >>> 0;
          }
        }

Caching Strategies

Cache-Aside (Lazy Loading)
Check cache first. Miss → fetch from DB, write to cache, return. Cache only holds requested data. Risk: cache stampede on cold start. Best general-purpose strategy.
Write-Through
Every write goes to cache AND DB synchronously. Always fresh. Downside: write latency doubles; unused writes waste cache space.
Write-Behind (Write-Back)
Write to cache immediately; DB write async/batched. Very low write latency. Risk: data loss if cache node fails before flush. Use for: metrics, counters.

Rate Limiting Algorithms

// Redis sliding window counter — production implementation
        async function isRateLimited(userId, limitPerMinute) {
          const now = Date.now();
          const currentWindow = Math.floor(now / 60000);
          const prevWindow = currentWindow - 1;
          const elapsed = (now % 60000) / 60000;
        
          const [prev, curr] = await redis.mget(`rl:${userId}:${prevWindow}`, `rl:${userId}:${currentWindow}`);
          const estimatedRate = parseInt(prev || '0') * (1 - elapsed) + parseInt(curr || '0');
        
          if (estimatedRate >= limitPerMinute) {
            return { limited: true, retryAfter: Math.ceil(60 - elapsed * 60) };
          }
          const pipe = redis.pipeline();
          pipe.incr(`rl:${userId}:${currentWindow}`);
          pipe.expire(`rl:${userId}:${currentWindow}`, 120);
          await pipe.exec();
          return { limited: false };
        }

Distributed Transactions: 2PC vs Saga

Two-Phase Commit (2PC)

Phase 1 (Prepare): coordinator asks all participants "can you commit?" Each locks resources. Phase 2 (Commit/Abort): if all say yes → commit, else abort. Problems: Coordinator is SPOF. Locks held during Phase 1 = blocking. Not suitable for microservices across orgs.

Saga Pattern

Sequence of local transactions. Each step publishes event to trigger next. On failure: compensating transactions undo completed steps. Choreography: services react to events (decoupled). Orchestration: central orchestrator (Step Functions, Temporal) — easier to reason about. Achieves eventual consistency, not ACID atomicity.

CQRS + Event Sourcing

Event Sourcing

Store sequence of events, not current state. State = replay events. Immutable, append-only log. Benefits: full audit trail, time-travel debugging, replay for new read models. Drawbacks: replay of millions of events is slow (use snapshots); schema evolution of historical events is hard.

CQRS

Separate write model (Commands) from read model (Queries). Read model = denormalized projection (materialized view, Elasticsearch, Redis). Updated async from write events. Combined with Event Sourcing: new read model = replay all events. Avoid for CRUD apps — complexity cost is high.

        const BASE62 = '0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz';
        function toBase62(num) {
          let result = '';
          while (num > 0) { result = BASE62[num % 62] + result; num = Math.floor(num / 62); }
          return result.padStart(7, '0');
        }
        
        // Redirect: Redis-first lookup, DB fallback, async click increment
        async function redirect(shortCode, redis, db) {
          const cached = await redis.get(`url:${shortCode}`);
          if (cached) {
            redis.incr(`clicks:${shortCode}`); // fire-and-forget
            return { status: 301, location: cached }; // 301=permanent (browser caches); 302 for tracking
          }
          const row = await db.findOne({ shortCode });
          if (!row) return { status: 404 };
          if (row.expires_at && row.expires_at < new Date()) return { status: 410 };
          await redis.setex(`url:${shortCode}`, 86400, row.long_url);
          return { status: 301, location: row.long_url };
        }

Twitter/Social Feed — Fanout Strategies

Fanout on Write (Push)

On post, push tweet to all follower timeline caches. Read is O(1). Celebrity problem: 10M followers = 10M writes per post.

Fanout on Read (Pull)

On timeline load, query tweets from all followees and merge. No write amplification. Problem: following 1K accounts = 1K queries per page load.

Hybrid (Twitter's Actual)

Push for regular users (<100K followers). Pull for celebrities. On read: merge pre-built feed with real-time celebrity tweets.

Interview Prep

System Design Interview — Q&A

How do you approach a system design interview in the first 5 minutes?
Never jump to components. Structure: (1) Clarify requirements — functional (what it does) and non-functional (scale, latency SLA, consistency, availability). (2) Back-of-envelope — QPS, storage, bandwidth. State assumptions explicitly. (3) Define the API — endpoints, request/response shapes. (4) High-level design — clients, LB, services, DB, cache, queues. (5) Deep dive — DB schema, sharding, consistency, failure modes. (6) Bottlenecks and trade-offs — proactively call out SPOFs. Senior differentiator: you drive the conversation, raise trade-offs before being asked, connect decisions to requirements.
CAP theorem in practice — how do you actually pick?
Ask: "What happens if a partition occurs — do I serve stale data or return an error?" If returning an error is worse (shopping cart, social feed, DNS) → AP. If stale data causes correctness violations (bank transfer, inventory deduction, leader election) → CP. In practice, most business logic is AP with application-level compensation: shopping cart merges on conflict, eventual inventory reconciliation. Lean AP and use idempotency + optimistic locking to handle edge cases rather than paying the latency cost of CP everywhere.
How do you handle "design this for 1 billion users"?
Don't design for 1B users on the whiteboard — design for the stated QPS and show scaling levers. Walk through each tier: stateless API servers (horizontal scale trivially), DB (the hard part — read replicas first, then sharding), cache (reduces DB load by 90%+), CDN (offloads static assets). Then say: "Given these numbers, the bottleneck is the database. I'd start with read replicas for the 10:1 read/write ratio, and introduce sharding once we hit the single-master write limit around ~10K writes/sec for Postgres." Show you can identify the actual bottleneck.
How do you communicate trade-offs effectively?
Use the format: "I'm choosing X over Y because [requirement Z]. The cost is [downside], which we can mitigate with [mitigation]." Example: "I'm choosing eventual consistency over strong consistency because our latency SLA is 50ms and strong consistency requires quorum reads which add ~20ms. The cost is users may briefly see stale data, which is acceptable for a social feed. We'll implement read-your-writes for the current user's own posts to prevent the jarring UX of posting and not seeing your own content."
How do you design for 99.99% uptime (four nines)?
99.99% = 52.6 minutes downtime/year. Requires: (1) Eliminate SPOFs — every component has redundancy (active-active or active-passive). (2) Multi-AZ deployment — AWS AZ failure must not take you down. (3) Circuit breakers — fail fast rather than queuing connections to a degraded service. (4) Graceful degradation — if recommendation service is down, show generic results, not 500. (5) Chaos engineering — regularly kill nodes in production to verify recovery. (6) Health checks + fast failover — LB removes unhealthy instances in <30s. (7) Zero-downtime deploys — blue/green or rolling. Five nines requires active-active multi-region — fundamentally harder.
How do you design idempotency at scale?
Idempotency key pattern: client generates UUID per logical operation, sends in Idempotency-Key header. Server: (1) check if key exists in idempotency store; (2) if exists, return cached response; (3) if not, process request, store (key → response) atomically with the operation; (4) expire keys after business-appropriate window. Atomicity is critical: store the idempotency record AND perform the operation in the same DB transaction — BEGIN; INSERT idempotency_records; do work; COMMIT. If INSERT fails (key already exists), return cached response. Hardest case: request times out, client retries with same key — idempotency record ensures second attempt returns same result, not duplicate.
Database migration without downtime — how?
Expand-contract pattern: Phase 1 (Expand) — add new column as nullable, deploy code that writes to BOTH old and new. Phase 2 — background-migrate existing rows in batches (1K rows with sleep between to avoid locking). Phase 3 — switch reads to new column. Phase 4 (Contract) — remove old column in separate deploy. For index creation: CREATE INDEX CONCURRENTLY in Postgres (no table lock). For large table column adds: ALTER TABLE ... ADD COLUMN in Postgres 11+ is instant for DEFAULT values. For table rewrites: use gh-ost (GitHub) or pg-osc — shadow table + incremental copy + atomic rename with ~1s lock.

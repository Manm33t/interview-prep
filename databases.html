Indexing — Internals & Strategies

B-Tree Index Internals

B-Trees (B+-Trees in most DBs) are the default index structure. All data rows are in leaf nodes linked in a doubly-linked list for range scans. Node size = one disk page (8 KB) = hundreds of keys. Depth = 3-4 levels for 1M rows. With OS page cache, hot pages stay in RAM = ~0.5ms total.

-- Composite index: column order is EVERYTHING
        CREATE INDEX idx_orders ON orders (user_id, status, created_at);
        -- Uses index: WHERE user_id = 5; WHERE user_id = 5 AND status = 'pending'
        -- Cannot use: WHERE status = 'pending' (no leading column)
        -- Rule: equality columns first, range column last, highest cardinality first
        
        -- Covering index: query satisfied entirely from index (no heap access = Index Only Scan)
        CREATE INDEX idx_orders_covering ON orders (user_id) INCLUDE (order_id, total, status);
        -- INCLUDE adds columns to leaf nodes only — not part of tree structure
        
        -- Partial index: only indexes rows matching WHERE clause
        CREATE INDEX idx_pending ON jobs (created_at) WHERE status = 'pending';
        -- Tiny index (0.1% of rows) — query MUST include the partial condition
        
        -- Expression index: index on function result
        CREATE INDEX idx_lower_email ON users (LOWER(email));
        -- Enables case-insensitive lookup via index: WHERE LOWER(email) = 'user@example.com'
        
        -- When indexes hurt:
        -- Write overhead: INSERT/UPDATE/DELETE updates ALL indexes → 10 indexes = 10× write amplification
        -- Low cardinality: boolean column has 2 values → planner prefers seq scan (50% match rate)
        -- Bloat: MVCC dead tuples accumulate → monitor with pgstattuple, REINDEX CONCURRENTLY
        
        -- Reading EXPLAIN ANALYZE:
        -- Seq Scan on large table = missing index
        -- "Rows Removed by Filter" = wasted I/O
        -- Buffers: hit=X read=Y → want high hit ratio (X / (X+Y))
        -- actual rows vs estimate divergence → run ANALYZE to update statistics

Database Design — Normalization, Sharding & Replication

Normalization Forms

1NF — Atomic values, no repeating groups, unique rows
Each column contains atomic (indivisible) values. No arrays stored in a single column. Fix: separate rows with a composite PK.
2NF — No partial dependencies on composite PK
Must be in 1NF. Every non-key column depends on the ENTIRE primary key. If product_name only depends on product_id (not the full composite PK of order_id + product_id), extract to a separate products table.
3NF — No transitive dependencies
Must be in 2NF. Non-key columns must depend on the key, not on other non-key columns. Classic example: city/state determined by zip_code (not by customer_id) → extract zip_codes table.
Denormalization trade-offs: write anomalies (must update all copies), storage cost, consistency risk. Always normalize first, denormalize only when EXPLAIN ANALYZE reveals the JOIN as the actual bottleneck.

Sharding Strategies

Range Sharding

Shard by key range. Supports range scans. Hotspot risk: new writes always hit latest shard.

Hash Sharding

hash(key) % N. Even distribution. Resharding problem: adding a shard requires rehashing ~1/N of all data. Use consistent hashing.

Directory Sharding

Lookup service maps entity → shard. Most flexible. SPOF: directory service must be cached aggressively.

OLTP vs OLAP

Dimension OLTP OLAP
Query patternPoint reads/writes, small row setsFull scans, aggregations, GROUP BY
SchemaNormalized (3NF)Star/snowflake schema — denormalized

Transactions, Isolation Levels & Concurrency

ACID Deep Dive

Atomicity

All-or-nothing. Implemented via Write-Ahead Log (WAL) — changes written to WAL first; on crash, incomplete transactions rolled back on recovery.

Isolation

Concurrent transactions execute as if serial. Tunable via isolation levels. Full isolation (Serializable) is expensive — most apps use Read Committed or Repeatable Read.

Durability

Committed data persists after crash. WAL fsync to durable storage. Never use fsync=off in production.

Consistency (ACID vs CAP)

ACID consistency = constraint enforcement (FK, unique, check). CAP consistency = replication freshness. These are different concepts — don't conflate them in interviews.

Isolation Levels

Level Dirty Read Non-Repeatable Phantom Use When
READ UNCOMMITTED✗✗✗Almost never. Postgres ignores it.
READ COMMITTED✓✗✗Default in Postgres, Oracle. Most OLTP.
REPEATABLE READ✓✓~*Default MySQL InnoDB. Financial reporting.
SERIALIZABLE✓✓✓Bank transfers, inventory, write-skew risk.

Optimistic vs Pessimistic Locking + MVCC

-- Pessimistic: lock the row on read
        SELECT * FROM inventory WHERE product_id = 99 FOR UPDATE;
        UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 99;
        COMMIT;
        
        -- Optimistic: version column — detect conflict on write
        UPDATE inventory
        SET quantity = quantity - 1, version = version + 1
        WHERE product_id = 99 AND version = 3;  -- 0 rows = conflict → retry
        -- Better when: conflicts are rare, read-heavy workloads, long read operations
        
        -- Deadlock prevention: always acquire locks in the same order
        SELECT * FROM accounts WHERE id IN (100, 200) ORDER BY id FOR UPDATE;
        -- ORDER BY ensures consistent lock order → deadlock eliminated
        
        -- MVCC (Postgres): each row has xmin/xmax (creating/deleting transaction IDs)
        -- Readers never block writers; writers never block readers.
        -- Dead rows accumulate → VACUUM reclaims space. Monitor: pg_stat_user_tables.n_dead_tup
        -- autovacuum must run regularly on high-churn tables or indexes bloat
Interview Prep

Database Interview — Deep Q&A

When do you choose SQL vs NoSQL?
Start with the data model. Relational with referential integrity, multi-entity transactions, ad-hoc query flexibility → SQL (Postgres). NoSQL when: (1) Document model — data is hierarchical, load whole document at once → MongoDB. (2) Key-value / session data — simple fast lookups → Redis, DynamoDB. (3) Write-heavy time-series — millions of writes/sec, append-only → Cassandra. (4) Graph traversal → Neo4j. The trap: "NoSQL scales better." Postgres with read replicas + connection pooling + proper indexing handles enormous scale. NoSQL trades ACID and query flexibility for specific performance characteristics.
What is the N+1 query problem and how do you fix it?
N+1 occurs when you fetch N parent records then make 1 additional query per parent — N+1 total instead of 2. Classic ORM lazy loading anti-pattern: fetch 100 orders, then for each fetch the user = 101 queries instead of 1 JOIN. Fixes: (1) SQL JOIN; (2) ORM eager loading (Prisma: include: { user: true }, TypeORM: relations: ['user']); (3) DataLoader pattern — collect all user_ids, single batch query SELECT * FROM users WHERE id = ANY($1). DataLoader is the standard GraphQL solution — batches and caches per request cycle.
How does connection pooling work and why is it critical?
Opening a new DB connection costs 20-100ms and ~5-10MB RAM on the DB server. Lambda at 1,000 req/sec creates 1,000 simultaneous connections — Postgres max_connections=100, so 900 requests get "connection refused." Connection pooling maintains idle connections and reuses them. For Lambda/serverless: use RDS Proxy or PgBouncer (transaction mode — DB connection held only during transaction, then returned). Pool sizing: (num_cores × 2) + effective_spindle_count per DB server. Monitor pool wait time and connection saturation separately.
How do you optimize a slow query?
Systematic: (1) EXPLAIN (ANALYZE, BUFFERS) — look for Seq Scans on large tables, high "Rows Removed by Filter", row estimate divergence (stale stats → run ANALYZE). (2) Missing indexes — WHERE clause columns without indexes on large tables. (3) Index usage — is planner choosing the right index? Stale stats can mislead it. (4) Implicit type casts — WHERE int_col = '5' may not use index. (5) JOIN type — Hash Join good for large datasets; Nested Loop good when inner set is small and indexed. (6) Large OFFSET — OFFSET 50000 scans 50K rows to skip; use keyset pagination: WHERE id > last_seen_id ORDER BY id LIMIT 20. (7) SELECT * — prevents covering indexes.
ACID vs BASE — and which NoSQL databases support which?
ACID: all-or-nothing transactions, constraint enforcement, full isolation, durable writes. Traditional relational DBs. MongoDB (single-document ACID; multi-doc transactions in v4+), DynamoDB (single-item ACID, transactions added). BASE: Basically Available, Soft state, Eventually consistent. Cassandra is BASE by design — tune consistency per query (ONE, QUORUM, ALL). DynamoDB default reads are eventually consistent but strongly consistent reads available at higher cost. Modern trend: many "NoSQL" DBs now offer optional transactions — the line is blurring.
Sharding vs partitioning vs replication — explain the difference.
Partitioning: divide a table into smaller pieces within the SAME DB server. Improves query performance (partition pruning), simplifies archival. No distribution. Sharding: horizontal partitioning across MULTIPLE database servers. Provides write and storage scalability. Introduces cross-shard query complexity and shard rebalancing. Replication: copies of the SAME data on multiple servers. Not about splitting data — about redundancy and read scaling. Primary handles writes, replicas serve reads. Provides HA (failover) and read scalability but NOT write scalability. They compose: sharded DB typically uses replication within each shard.
PostgreSQL vs MongoDB — trade-offs a senior engineer must know.
Postgres strengths: full ACID, rich SQL (CTEs, window functions, lateral joins), enforced schema, PostGIS, JSONB for semi-structured data. Well-understood scaling path. MongoDB strengths: flexible schema (no migrations for field additions), native document model avoids impedance mismatch, horizontal scale via sharding built-in, Atlas Search (full-text without Elasticsearch). Postgres JSONB vs MongoDB: Postgres JSONB supports GIN indexes + full SQL alongside document. For truly document-centric data with high write volume and schema churn, MongoDB's change streams and TTL indexes are genuine advantages. Real answer: start with Postgres, migrate to MongoDB only when you hit a specific limitation it can't solve.
How do you perform schema migration with zero downtime?
Expand-contract across multiple deployments: Step 1 (Expand) — add new column as nullable, deploy code writing to both old and new. Step 2 — backfill existing rows in batches (1K rows at a time + sleep to avoid locking). Step 3 — switch reads to new column after backfill. Step 4 (Contract) — remove old column. For index creation: CREATE INDEX CONCURRENTLY (no table lock, takes longer). For large table column adds in Postgres 11+: ADD COLUMN with DEFAULT is instant. For table rewrites: gh-ost (GitHub) or pg-osc — shadow table, incremental copy, atomic rename with ~1s lock.

PostgreSQL Internals

PostgreSQL is the most interview-relevant RDBMS for senior roles. Understanding its internals — heap files, B-tree index structure, MVCC, VACUUM, WAL — puts you in the top 5% of candidates at companies like DEEL, Stripe, and Shopify.

How PostgreSQL Stores Data

-- MVCC: each row has xmin (created by) and xmax (deleted by) transaction IDs
        -- A row is visible if: xmin committed AND (xmax is null OR xmax not committed)
        -- This allows readers and writers to never block each other
        
        -- Dead tuples: after UPDATE, old version lingers until VACUUM reclaims it
        -- High UPDATE/DELETE rate → table bloat → slow seq scans
        -- Monitor bloat:
        SELECT schemaname, tablename,
          n_live_tup, n_dead_tup,
          round(n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 2) AS dead_pct,
          last_autovacuum, last_autoanalyze
        FROM pg_stat_user_tables
        ORDER BY n_dead_tup DESC;
        
        -- WAL (Write-Ahead Log): every change is first written to WAL before the heap
        -- This enables: crash recovery, replication (streaming + logical), point-in-time recovery
        -- wal_level = logical → required for logical replication / CDC (Debezium, etc.)
        
        -- Connection pooling is critical: Postgres spawns a process per connection
        -- 200 connections × 5MB/process = 1GB RAM just for idle connections
        -- Use PgBouncer (transaction mode) for serverless/high-concurrency workloads
        
        -- Check connection usage:
        SELECT count(*), state FROM pg_stat_activity GROUP BY state;
        
        -- Replication lag (for read replicas):
        SELECT now() - pg_last_xact_replay_timestamp() AS replication_lag;

Index Types & When to Use Each

B-Tree (default)

Equality, range (<, >, BETWEEN), ORDER BY, LIKE 'prefix%'. 99% of use cases. Supports multi-column (composite) indexes.

Hash

Equality only (=). O(1) lookup vs B-tree's O(log n). Not replicated before PG10. Rarely worth switching from B-tree.

GIN (Generalized Inverted)

Full-text search (tsvector), JSONB containment (@>), array overlap (&&). Slower to build, fast to query many keys per row.

GiST

Geometric/geographic data (PostGIS), range types, fuzzy string search (pg_trgm). Lossy — may return false positives, rechecked.

Partial Index

WHERE status = 'pending' — indexes only matching rows. Tiny, fast, avoids indexing large cold data.

Covering Index (INCLUDE)

CREATE INDEX ON t(a) INCLUDE (b,c) — stores extra cols in leaf pages. Enables index-only scans — zero heap access.

Query Planning & EXPLAIN ANALYZE

Reading query plans is a non-negotiable skill at DEEL and Stripe-level companies. Every slow query in production has a plan that explains why it's slow.

-- Always use EXPLAIN (ANALYZE, BUFFERS) — not just EXPLAIN
        -- EXPLAIN shows estimated plan; ANALYZE actually runs it and shows actual times
        EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
        SELECT u.name, count(o.id)
        FROM orders o JOIN users u ON o.user_id = u.id
        WHERE o.status = 'active'
        GROUP BY u.name;
        
        -- Key plan nodes to recognize:
        -- Seq Scan     → reading entire table. BAD on large tables. Fix: add index.
        -- Index Scan   → B-tree traversal + heap fetch per row. Good for selective queries.
        -- Index Only Scan → no heap fetch (covering index). BEST for read-heavy.
        -- Bitmap Heap Scan → batches ctids, sorts by physical location, one heap pass. Good for 1-5% selectivity.
        -- Hash Join    → build hash table from smaller relation, probe with larger. Good for equi-joins.
        -- Nested Loop  → for each outer row, scan inner. O(n*m). Good only when inner is indexed.
        -- Merge Join   → both inputs sorted on join key. O(n+m). Good with sorted inputs.
        
        -- Reading cost: (start_cost..total_cost) — planner units, not seconds
        -- rows= is the planner ESTIMATE; actual rows= is measured
        -- LARGE difference between estimated and actual rows → stale statistics → run ANALYZE
        
        -- Fix slow queries checklist:
        -- 1. Is the filter column indexed?
        -- 2. Is the index being used? (EXPLAIN output)
        -- 3. Is the index selective enough? (e.g., boolean index rarely helps)
        -- 4. Composite index column order correct? (equality cols first, then range)
        -- 5. Are statistics fresh? (last_analyze in pg_stat_user_tables)
        -- 6. Is the query doing a LIKE '%suffix%'? (needs pg_trgm GIN index)
        -- 7. Does JOIN order matter? (use SET enable_nestloop=off to test)
        
        -- Force index use (testing only — let planner decide in production):
        SET enable_seqscan = off;
        
        -- Update statistics after large data loads:
        ANALYZE orders;
        VACUUM ANALYZE orders;  -- also reclaims dead tuples

DB Expert Q&A

Database Expert Interview Q&A

DEEL asks: "You have a table with 100M rows. A query that runs in 5ms starts taking 30 seconds. What do you do?"
Step 1: Run EXPLAIN (ANALYZE, BUFFERS) on the slow query. Look for Seq Scan where there should be an Index Scan, high "rows removed by filter" (low selectivity), or large difference between estimated and actual rows (stale stats). Step 2: Check pg_stat_user_tables for last_autovacuum and n_dead_tup — table bloat from dead tuples forces Seq Scan even with indexes. Run VACUUM ANALYZE. Step 3: Check for lock contention with pg_stat_activity — the query might be waiting, not running slowly. Step 4: Look for the trigger: was there a schema change, a new deployment that changed the query, a data skew (e.g., one user_id now has 10M rows breaking index selectivity), or table statistics that failed to update after a large bulk import? Step 5: For the fix: add missing index, fix composite index column order (ESR rule), run ANALYZE to refresh statistics, or rewrite the query to avoid non-SARGable predicates like WHERE LOWER(email) = ? (use a functional index instead).
Explain MVCC in PostgreSQL. Why can a long-running transaction hurt performance?
MVCC (Multi-Version Concurrency Control) means every UPDATE creates a new row version — the old version is kept as a "dead tuple" until VACUUM reclaims it. This allows readers and writers to never block each other: readers see a consistent snapshot based on their transaction start time, writers write new versions. A long-running transaction holds back the "oldest transaction" horizon — VACUUM cannot reclaim dead tuples that might still be visible to that transaction. The result: table bloat grows unboundedly during that transaction. A single idle transaction open for hours can cause every other UPDATE/DELETE to accumulate dead tuples, leading to Seq Scans overtaking Index Scans and 10×-100× query slowdowns. Fix: set statement_timeout and idle_in_transaction_session_timeout, monitor pg_stat_activity for long-running transactions, use connection pooling (PgBouncer) to detect idle connections.
When would you use a partial index, and how does it differ from a composite index?
A partial index has a WHERE clause that limits which rows are indexed. Use it when: (1) You query a small subset of rows frequently (e.g., WHERE status = 'pending' — only 1% of the table). The index is tiny, fits in memory, and is faster than indexing all rows. (2) You want a unique constraint on a subset: CREATE UNIQUE INDEX ON orders(email) WHERE deleted_at IS NULL. A composite (multi-column) index covers multiple columns in the predicate. Column order matters — put equality conditions first, then range. Example: (user_id, created_at) efficiently handles WHERE user_id = ? AND created_at > ? because the index first narrows to user_id (equality, high cardinality) then range-scans created_at. They can be combined: CREATE INDEX ON orders(user_id, created_at) WHERE status = 'active' — a partial composite index that's both selective and multi-column.
What is the N+1 problem and how do you solve it in both SQL and MongoDB contexts?
N+1 occurs when you fetch N parent records and then issue N individual queries for related records — total: N+1 queries. In SQL: you query for 100 users, then for each user issue SELECT * FROM posts WHERE user_id = ? — 101 queries instead of 1. Fix: eager loading with JOIN or ORM-level includes (Sequelize: include, TypeORM: relations eager loading). In MongoDB/Node (Mongoose): you fetch 100 orders, then for each do User.findById(order.userId) — 101 queries. Fix: populate (Mongoose), $lookup aggregation, or manual batch lookup using $in: User.find({ _id: { $in: orderUserIds } }) then merge in application code — 2 queries total. In GraphQL: the N+1 problem is endemic because each resolver is independent. Fix: DataLoader (Facebook) — batches all individual lookups within a single event-loop tick into one batch query, then distributes results back to individual resolvers.
Explain database sharding. What are the pitfalls?
Sharding horizontally partitions data across multiple database instances (shards). Three strategies: (1) Range sharding — shard by date or ID range. Simple, but leads to hot spots (all recent writes go to one shard). (2) Hash sharding — shard_id = hash(key) % N. Uniform distribution, but range queries span all shards. (3) Directory-based sharding — lookup table maps key to shard. Most flexible, but lookup table becomes a bottleneck/single point of failure. Pitfalls: Cross-shard JOINs require scatter-gather (query all shards, merge in app layer) — expensive. Cross-shard transactions require distributed 2PC — complex and slow. Re-sharding when a shard fills up is operationally complex. Hotspot shards (e.g., a large tenant in a multi-tenant SaaS). Application code must be shard-aware unless using middleware (Vitess, Citus). Most teams should exhaust vertical scaling, read replicas, and caching before sharding.

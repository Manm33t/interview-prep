Node.js

What is Node.js?

Node.js is a single-threaded, non-blocking, event-driven JavaScript runtime built on Chrome's V8 engine. It uses libuv for cross-platform async I/O and an event loop to handle concurrency without threads.

Your Application Code (JS)
↕
Node.js APIs (fs, http, crypto, events, streams…)
↕
V8 Engine (JavaScript execution + JIT compilation)
↕
libuv (Event loop, Thread pool, OS async I/O)
↕
Operating System (Kernel, Network, File System)

Why Node.js?

⚡

Non-blocking I/O

Can handle thousands of concurrent connections without spawning threads. Perfect for I/O-heavy workloads (APIs, databases, file ops).

🔗

Single Language

JavaScript on both frontend and backend. Share code, types, and validation logic (e.g., Zod schemas) across the stack.

📦

npm Ecosystem

World's largest package registry with 2M+ packages. Huge community, rich tooling.

🚀

V8 Performance

Google's V8 JIT-compiles JS to native machine code, making Node.js fast enough for many server-side workloads.

When NOT to use Node.js: CPU-intensive tasks (video encoding, ML inference, complex number crunching). Node's single main thread means heavy computation blocks all other requests. Use Worker Threads or delegate to a separate service.

The Event Loop (Deep Dive)

The event loop is the mechanism that allows Node.js to perform non-blocking I/O despite JavaScript being single-threaded. Understanding it is critical for writing performant Node code and acing interviews.

Event Loop Phases

1
timers

Executes callbacks scheduled by setTimeout and setInterval (when threshold has elapsed)

↓
2
pending callbacks

Executes I/O callbacks deferred from previous loop iteration

↓
3
idle, prepare

Internal libuv use only

↓
4
poll

Retrieves new I/O events; executes I/O callbacks. Node.js will block here if nothing is scheduled.

↓
5
check

Executes setImmediate callbacks

↓
6
close callbacks

Executes close event callbacks (e.g., socket.on("close", ...))

Microtask Queue (Highest Priority)

Between each phase, Node drains the microtask queue completely before moving to the next phase:

// Microtask priority order:
        // 1. process.nextTick callbacks (highest priority!)
        // 2. Promise callbacks (.then / .catch / .finally)
        
        // Demo
        console.log("1 — synchronous");
        
        setTimeout(() => console.log("2 — setTimeout"), 0);
        setImmediate(() => console.log("3 — setImmediate"));
        
        Promise.resolve().then(() => console.log("4 — Promise.then"));
        process.nextTick(() => console.log("5 — nextTick"));
        
        console.log("6 — synchronous");
        
        // Output:
        // 1 — synchronous
        // 6 — synchronous
        // 5 — nextTick          ← microtask queue (nextTick first)
        // 4 — Promise.then      ← microtask queue (Promise second)
        // 2 — setTimeout        ← timers phase
        // 3 — setImmediate      ← check phase

setTimeout vs setImmediate

// Inside I/O callback — setImmediate ALWAYS runs before setTimeout
        const fs = require("fs");
        fs.readFile("file.txt", () => {
          setTimeout(() => console.log("timeout"), 0);
          setImmediate(() => console.log("immediate"));
          // Always: "immediate" before "timeout"
        });
        
        // Outside I/O callback — ORDER IS NOT GUARANTEED
        // (depends on process startup time and OS scheduling)
        setTimeout(() => console.log("timeout"), 0);
        setImmediate(() => console.log("immediate"));
        // Non-deterministic: either order possible

Don't Block the Event Loop!

// ❌ BAD: synchronous operation blocks all requests
        app.get("/heavy", (req, res) => {
          const result = heavyCPUCalc(); // blocks event loop for entire duration!
          res.json(result);
        });
        
        // ✅ OPTION 1: Worker Thread for CPU-heavy work
        const { Worker, isMainThread, parentPort } = require("worker_threads");
        
        app.get("/heavy", (req, res) => {
          const worker = new Worker("./heavyCalc.js", { workerData: req.body });
          worker.on("message", (result) => res.json(result));
          worker.on("error", (err) => res.status(500).json({ error: err.message }));
        });
        
        // ✅ OPTION 2: setImmediate to yield periodically
        function processInChunks(data, chunkSize = 1000) {
          let i = 0;
          function processChunk() {
            const end = Math.min(i + chunkSize, data.length);
            while (i < end) { process(data[i++]); }
            if (i < data.length) setImmediate(processChunk); // yield to event loop
          }
          processChunk();
        }

libuv Thread Pool

Thread Pool (default: 4 threads): libuv uses a thread pool for operations that don't have OS-level async support: fs.* file operations, crypto (hash, pbkdf2), DNS lookup (dns.lookup), and some compression. Network I/O (TCP, HTTP) uses OS-level async — no thread pool needed. Set UV_THREADPOOL_SIZE=8 env var to increase pool size (max 128).

Streams & Buffers

Streams allow processing data piece-by-piece instead of loading everything into memory. Essential for large files, HTTP responses, and real-time data.

Four Stream Types

📖

Readable

Data source: fs.createReadStream, http.IncomingMessage

✏️

Writable

Data destination: fs.createWriteStream, http.ServerResponse

🔄

Duplex

Both readable and writable: net.Socket

⚙️

Transform

Modifies data as it passes through: zlib.createGzip()

Using Streams

const { pipeline } = require("stream/promises"); // Node 15+
        const zlib = require("zlib");
        
        // Compress a large file without loading it all into memory
        async function compressFile(input, output) {
          await pipeline(
            fs.createReadStream(input),
            zlib.createGzip(),
            fs.createWriteStream(output)
          );
        }
        
        // Custom Transform stream
        const { Transform } = require("stream");
        
        class CSVParser extends Transform {
          constructor() {
            super({ objectMode: true });
            this.headers = null;
            this.buffer = "";
          }
        
          _transform(chunk, encoding, callback) {
            this.buffer += chunk.toString();
            const lines = this.buffer.split("\n");
            this.buffer = lines.pop(); // save incomplete line
        
            for (const line of lines) {
              if (!this.headers) {
                this.headers = line.split(",").map(h => h.trim());
              } else {
                const values = line.split(",");
                const row = Object.fromEntries(this.headers.map((h, i) => [h, values[i]]));
                this.push(row);
              }
            }
            callback();
          }
        
          _flush(callback) {
            if (this.buffer && this.headers) {
              const values = this.buffer.split(",");
              this.push(Object.fromEntries(this.headers.map((h, i) => [h, values[i]])));
            }
            callback();
          }
        }
        
        // Backpressure — key concept
        const readable = fs.createReadStream("big.csv");
        const writable = fs.createWriteStream("out.csv");
        
        readable.on("data", (chunk) => {
          const canContinue = writable.write(chunk);
          if (!canContinue) {
            readable.pause(); // backpressure: slow down source
            writable.once("drain", () => readable.resume()); // resume when buffer drains
          }
        });

        async function verifyPassword(password, storedHash) {
          const [saltHex, hashHex] = storedHash.split(":");
          const salt = Buffer.from(saltHex, "hex");
          const key = await new Promise((res, rej) =>
            crypto.scrypt(password, salt, 64, (err, k) => err ? rej(err) : res(k))
          );
          return crypto.timingSafeEqual(key, Buffer.from(hashHex, "hex"));
        }
        
        // ── Symmetric encryption (AES-256-GCM) ────────────────
        function encrypt(text, key) {
          const iv = crypto.randomBytes(12);
          const cipher = crypto.createCipheriv("aes-256-gcm", key, iv);
          const encrypted = Buffer.concat([cipher.update(text, "utf8"), cipher.final()]);
          const tag = cipher.getAuthTag();
          return { iv: iv.toString("hex"), encrypted: encrypted.toString("hex"), tag: tag.toString("hex") };
        }
        
        function decrypt({ iv, encrypted, tag }, key) {
          const decipher = crypto.createDecipheriv("aes-256-gcm", key, Buffer.from(iv, "hex"));
          decipher.setAuthTag(Buffer.from(tag, "hex"));
          return decipher.update(Buffer.from(encrypted, "hex")) + decipher.final("utf8");
        }

Child Processes

The child_process module spawns subprocesses — run shell commands, other Node scripts, or any binary. Unlike Worker Threads, child processes have separate memory and a separate V8 instance.

const { exec, execFile, spawn, fork } = require("child_process");
        const { promisify } = require("util");
        const execAsync = promisify(exec);
        
        // ── exec: run shell command, buffer output ─────────────
        // ⚠️ shell injection risk if you interpolate user input!
        const { stdout, stderr } = await execAsync("git log --oneline -5");
        console.log(stdout);
        
        // ── execFile: safer — no shell, args passed separately ──
        const { stdout: out } = await promisify(execFile)("git", ["log", "--oneline", "-5"]);
        
        // ── spawn: streaming I/O (large output, long-running) ───
        const ls = spawn("ls", ["-la", "/usr/local"]);
        ls.stdout.on("data", chunk => process.stdout.write(chunk));
        ls.stderr.on("data", chunk => process.stderr.write(chunk));
        ls.on("close", code => console.log(`exited: ${code}`));
        
        // ── fork: spawn another Node.js script, with IPC channel ─
        const child = fork("./worker.js", ["--env", "production"]);
        
        child.send({ type: "PROCESS", payload: bigData });  // send message to child

        // Worker Threads → CPU-bound JS that shares memory (faster, lighter than fork)

                () => new Worker(workerPath));
              this.queue = [];
            }

              return new Promise((resolve, reject) => {
                const worker = this.workers.find(w => w.state === "idle");
                if (worker) {
                  this.execute(worker, data, resolve, reject);
                } else {
                  this.queue.push({ data, resolve, reject });
                }
              });

            return data.map(x => x * x).reduce((a, b) => a + b, 0);

          parentPort.postMessage(result);

Cluster Module

Node.js runs on a single CPU core by default. The cluster module forks worker processes so you can use all cores on a machine, each handling incoming connections.

const cluster = require("cluster");
        const os = require("os");
        const http = require("http");
        
        const NUM_WORKERS = os.cpus().length;
        
        if (cluster.isPrimary) {
          console.log(`Primary PID ${process.pid} — forking ${NUM_WORKERS} workers`);
        
          // Fork one worker per CPU core
          for (let i = 0; i < NUM_WORKERS; i++) {
            cluster.fork();
          }
        
          // Auto-restart crashed workers
          cluster.on("exit", (worker, code, signal) => {
            console.warn(`Worker ${worker.process.pid} died (${signal || code}). Restarting...`);
            cluster.fork();
          });
        
          // Zero-downtime rolling restart (send SIGUSR2)
          process.on("SIGUSR2", () => {
            const workers = Object.values(cluster.workers);
            let i = 0;
            const restartNext = () => {
              if (i >= workers.length) return;
              const worker = workers[i++];
              worker.disconnect();
              worker.on("exit", () => {
                cluster.fork().on("listening", restartNext);
              });
            };
            restartNext();
          });
        
        } else {
          // Each worker runs this — share the same port
          http.createServer((req, res) => {
            res.end(`Handled by worker PID ${process.pid}\n`);
          }).listen(3000);
        
          console.log(`Worker PID ${process.pid} started`);
        }
        
        // ── Cluster vs Worker Threads ─────────────────────────
        // Cluster:        separate processes, no shared memory, multi-core HTTP servers
        // Worker Threads: same process, shared memory (SharedArrayBuffer), CPU-bound JS
        // PM2:            production-grade process manager — prefer over manual clustering
        // pm2 start app.js -i max   # auto-forks to all cores with monitoring + restarts
Production tip: In most cases prefer PM2 (pm2 start server.js -i max) or a container orchestrator (K8s) over writing cluster code manually. They handle restarts, logs, monitoring, and zero-downtime reloads out of the box.

// Middleware = function(req, res, next) — chain of responsibility
        
        // 1. Authentication middleware
        const authenticate = async (req, res, next) => {
          const token = req.headers.authorization?.split(" ")[1];
          if (!token) return res.status(401).json({ error: "No token" });
          try {
            req.user = await verifyJwt(token);
            next();
          } catch {
            res.status(401).json({ error: "Invalid token" });
          }
        };
        
        // 2. Authorization middleware factory
        const authorize = (...roles) => (req, res, next) => {
          if (!roles.includes(req.user?.role)) {
            return res.status(403).json({ error: "Forbidden" });
          }
          next();
        };
        
        // 3. Validation middleware with Zod
        const validate = (schema) => (req, res, next) => {
          const result = schema.safeParse(req.body);
          if (!result.success) {
            return res.status(400).json({ errors: result.error.flatten() });
          }
          req.validatedBody = result.data;
          next();
        };
        
        // 4. Rate limiting
        const rateLimit = (maxReq, windowMs) => {
          const counters = new Map();
          return (req, res, next) => {
            const key = req.ip;
            const now = Date.now();
            const record = counters.get(key) ?? { count: 0, resetAt: now + windowMs };
        
            if (now > record.resetAt) {
              record.count = 0;
              record.resetAt = now + windowMs;
            }
        
            record.count++;
            counters.set(key, record);
        
            if (record.count > maxReq) {
              return res.status(429).json({ error: "Too many requests" });
            }
            next();
          };
        };
        
        // Usage
        app.post(
          "/api/users",
          rateLimit(100, 60_000),
          authenticate,

Node.js uses V8's garbage collector. The heap is divided into Young (short-lived objects) and Old (long-lived objects) generations. Understanding this helps you write GC-friendly code and diagnose leaks.

V8 Heap Structure

🌱

Young Generation

New objects allocated here (fast minor GC). Most objects die young — collected in milliseconds via Scavenge.

🏛️

Old Generation

Survivors promoted here. Major GC (Mark-Sweep) is slower — avoid promoting too many objects.

💾

External Memory

Buffers and TypedArrays live outside the V8 heap — tracked separately in memoryUsage().external.

📊

RSS

Resident Set Size — total memory the process holds in RAM including heap, stack, and external buffers.

            this.state = "CLOSED";
            this.nextRetry = null;
          }
        
          async call(...args) {
            if (this.state === "OPEN") {
              if (Date.now() < this.nextRetry) throw new Error("Circuit OPEN");
              this.state = "HALF_OPEN";
            }
            try {
              const result = await this.fn(...args);
              this.onSuccess();
              return result;
            } catch (err) {
              this.onFailure();
              throw err;
            }
          }
        
          onSuccess() {
            this.failures = 0;
            this.state = "CLOSED";
          }
        
          onFailure() {
            this.failures++;
            if (this.failures >= this.threshold) {
              this.state = "OPEN";
              this.nextRetry = Date.now() + this.timeout;
            }
          }
        }

Node.js Interview Q&A

Q: How does the Node.js event loop work?
Node.js runs JavaScript on a single thread. The event loop (powered by libuv) cycles through 6 phases: (1) timers, (2) pending callbacks, (3) idle/prepare, (4) poll, (5) check, (6) close callbacks. Between each phase, the microtask queue is fully drained — first process.nextTick, then Promise callbacks. I/O callbacks from network/file operations complete asynchronously via the OS or libuv's thread pool, then their callbacks are pushed to the event loop. This allows handling thousands of concurrent I/O operations without blocking.
Q: What is the difference between process.nextTick and setImmediate?
process.nextTick runs before the next event loop iteration — immediately after the current operation, draining the nextTick queue completely. setImmediate runs in the check phase of the current/next event loop iteration. nextTick has higher priority and can starve I/O if called recursively (infinite nextTick loop = nothing else runs). Prefer setImmediate for breaking up long operations to allow I/O callbacks to run.
Q: What is the libuv thread pool and what does it handle?
libuv maintains a thread pool (default 4, max 128 via UV_THREADPOOL_SIZE) for operations that can't use OS-level async APIs: fs.* file operations, crypto (pbkdf2, scrypt, randomBytes), dns.lookup, and zlib. Network I/O (TCP/UDP) uses OS-level async (epoll/kqueue/IOCP) and does NOT use the thread pool. This is why reading many files concurrently can be faster with a larger thread pool.
Q: How do you handle CPU-intensive tasks in Node.js?
Four strategies: (1) Worker Threads (Node 12+) — run JS in separate threads sharing memory via SharedArrayBuffer; best for CPU-bound JS. (2) Child Processes (child_process.fork) — spawn separate Node.js process; heavier but fully isolated. (3) Native addons (C++) — maximum performance; complex to write. (4) Offload to a separate service — queue CPU work to a dedicated microservice (Python for ML, Go for compute). Never run heavy CPU work on the main thread in production.
Q: What are streams and why use them?
Streams process data in chunks instead of loading everything into memory. A 1GB file loaded with fs.readFile takes 1GB of RAM; a read stream processes it in ~64KB chunks. Four types: Readable, Writable, Duplex, Transform. Key concept: backpressure — when the writable side is slower than the readable side, the stream signals to pause the source. Use pipeline() from stream/promises — it handles backpressure and error propagation automatically.
Q: Explain the module caching mechanism in Node.js.
After require() loads a module for the first time, Node.js caches the resolved exports object in require.cache keyed by the absolute file path. Subsequent require() calls for the same module return the cached object synchronously — the module code does NOT re-execute. This makes singletons work naturally (e.g., a single DB connection instance). You can clear the cache with delete require.cache[require.resolve('./module')] — useful in tests.
Q: What is clustering in Node.js and when would you use it?
The cluster module forks multiple worker processes, each running the same server code. The master process distributes incoming connections to workers (round-robin on most OSes). Use it to utilize all CPU cores on a multi-core machine — a Node server can only use one core per process otherwise. Workers share no memory (unlike Worker Threads). In production, prefer PM2 or container orchestration (K8s) over manual clustering. Stateful sessions break clustering — use Redis for shared session state.
Q: What causes memory leaks in Node.js?
Common causes: (1) Uncleared timers/intervals — setInterval holds references; always call clearInterval. (2) EventEmitter listeners not removed — each on() adds a listener; not removing them is a leak. (3) Global variables accumulating data — every global persists for process lifetime. (4) Closures holding large objects unexpectedly. (5) Caches without eviction — unbounded Maps/objects grow forever. (6) Circular references in non-WeakMap structures. Tools: process.memoryUsage(), Node.js heap snapshots in Chrome DevTools, or --inspect flag.
Q: How does Node.js handle errors in async code?
Three layers: (1) try/catch with async/await — catches rejected promises in async functions. (2) .catch() on Promise chains. (3) Process-level handlers: process.on('unhandledRejection', ...) for unhandled promise rejections, and process.on('uncaughtException', ...) for synchronous throws that escape all handlers. Always exit the process after uncaughtException — the process is in an unknown state. In Express, always call next(err) in async routes and have a 4-parameter error handler.
Q: What is the difference between CommonJS and ESM?
CJS: synchronous require(), dynamic (can be inside conditionals), module.exports. Analyzed at runtime. ESM: asynchronous static import statements, only at top level (unless using import() dynamic import), export keyword. Analyzed at parse time — enables tree-shaking. ESM has live bindings (exported values update), CJS copies the value. Interop: CJS can require() CJS; ESM can import CJS (gets default export); CJS cannot natively require() ESM (must use dynamic import()).
Q: How do you implement graceful shutdown in a Node.js server?
Listen for SIGTERM (Docker/K8s sends this) and SIGINT (Ctrl+C). On signal: (1) stop accepting new connections (server.close()); (2) wait for in-flight requests to complete (track with a counter or use a library like stoppable); (3) close database connections; (4) flush queues/caches; (5) call process.exit(0). Set a timeout (e.g., 30s) to force exit if shutdown takes too long. This ensures zero dropped requests and clean resource release during deployments.
Q: What is backpressure in Node.js streams?
Backpressure is the mechanism to signal a data producer to slow down when the consumer can't keep up. In Node.js streams, writable.write(chunk) returns false when its internal buffer is full. The readable side should call readable.pause() and resume on the writable's 'drain' event. If ignored, the internal buffer grows unboundedly, causing memory exhaustion. Using pipeline() handles this automatically — this is the main reason to use it over manual pipe().

System Design Q&A

Q: How would you scale a Node.js API to handle 100k requests/second?
Horizontal scaling strategy: (1) Stateless API — no server-side session state; use JWTs or Redis for sessions. (2) Load balancer (Nginx, AWS ALB) distributing across multiple Node.js instances. (3) Database connection pooling — limit connections, use read replicas for read-heavy workloads. (4) Caching layer — Redis for hot data, CDN for static assets; cache-aside or read-through patterns. (5) Queue async work — SQS/RabbitMQ for background jobs, don't do heavy work in request path. (6) Use cluster module or PM2 to use all CPU cores per machine. (7) Monitor and autoscale — CloudWatch/K8s HPA based on CPU/request latency.
Q: Explain event-driven architecture vs request-response.
Request-response: caller waits synchronously for a result (HTTP API). Tight coupling — caller must know where to send the request. EDA: services communicate via events on a bus (EventBridge, Kafka). Publisher emits an event; subscribers react independently. Benefits: loose coupling, fan-out (one event → many consumers), temporal decoupling (consumer can be down), replay capability. Tradeoffs: harder to debug end-to-end flows, eventual consistency, idempotency required (events may be delivered more than once). Best for: workflows where steps can happen in parallel, or where source service shouldn't know about consumers.
Q: How do you ensure idempotency in a distributed system?
Idempotency means an operation can be applied multiple times with the same result. Strategies: (1) Idempotency keys — client sends a unique key per logical operation; server stores results by key and returns cached result on duplicate. (2) Content-based deduplication — hash the request payload; reject/return cached result if hash seen before. (3) Database constraints — unique index on natural key prevents duplicate inserts. (4) Conditional writes — check-and-set, optimistic locking with version numbers. Critical for: payment processing, fax sending, email delivery, any at-least-once delivery system.
Q: What is the circuit breaker pattern and when would you use it?
Circuit breaker protects against cascading failures when a dependency (external API, database) is degraded. Three states: CLOSED (normal), OPEN (fail-fast, no calls to dependency), HALF-OPEN (test probe). Transitions: CLOSED → OPEN after N consecutive failures; OPEN → HALF-OPEN after timeout; HALF-OPEN → CLOSED on success, back to OPEN on failure. Benefits: (1) prevents request pile-up on a slow dependency from exhausting your thread pool; (2) gives failing service time to recover; (3) provides clear operational visibility. Use for: external payment processors, third-party APIs, slow database connections, any remote call with potential for degradation.
Q: How would you design a rate limiter?
Four algorithms: (1) Fixed window — count requests per time window; simple but allows burst at window boundary. (2) Sliding window log — store timestamps of each request; more accurate, higher memory. (3) Sliding window counter — blend of fixed windows, approximated; good balance. (4) Token bucket — add tokens at fixed rate, deduct on request; allows controlled bursting. Implementation in distributed systems: use Redis with atomic operations (INCR + EXPIRE, or Lua scripts for atomic check-and-increment). Key by: IP address (DDoS protection), user ID (per-customer limits), or API key.
Q: How do you handle database transactions in Node.js + MongoDB?
MongoDB 4.x+ supports multi-document ACID transactions via sessions. Start a session, begin a transaction, run operations, then commit or abort. In Mongoose: use session.withTransaction() — it auto-retries on transient errors. Caveats: transactions require a replica set or sharded cluster (not available on standalone MongoDB). For most cases, design schemas to keep related data in a single document (subdocuments) and use atomic operations like $push, $inc, $set — these are always atomic on a single document without transactions.

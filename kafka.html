Kafka Fundamentals

Apache Kafka is a distributed, fault-tolerant, high-throughput event streaming platform. It decouples producers from consumers, persists events durably, and enables replay — unlike traditional message queues. Understanding Kafka internals is expected for senior distributed systems roles.

Topic & Partition

A topic is a named stream. Partitions enable parallelism — each partition is an ordered, immutable log. More partitions = more consumer parallelism, but more overhead.

Offset

Each message has a monotonically increasing offset per partition. Consumers track their offset — they can replay from any point. Kafka retains messages by time or size, not until consumed.

Consumer Group

Multiple consumer groups can read the same topic independently. Within a group, each partition is assigned to exactly one consumer — enables horizontal scaling. Max parallelism = num partitions.

Replication

Each partition has one leader and N-1 followers. Producers write to leader; followers replicate. ISR (In-Sync Replicas) — replicas fully caught up. acks=all → wait for all ISR to acknowledge.

Retention

Default: 7 days or 1GB per partition. Compacted topics: retain only the last value per key (useful for maintaining current state). Events are not deleted on consumption.

ZooKeeper → KRaft

Pre-3.0: ZooKeeper managed cluster metadata. Post-3.0: KRaft mode — Kafka manages its own metadata via Raft consensus. No ZooKeeper dependency in new deployments.

Kafka Internals — Why It's Fast

// Kafka's speed comes from 4 design decisions:
        
        // 1. Sequential disk I/O — append-only log
        //    Random disk I/O: ~100 ops/sec (seek time dominates)
        //    Sequential disk I/O: ~1M ops/sec (no seek, OS prefetch)
        //    Kafka always appends to the end of a partition file → sequential writes
        
        // 2. Zero-copy — sendfile() syscall
        //    Normal path: disk → kernel buffer → user buffer → socket buffer → NIC (4 copies)
        //    Zero-copy:   disk → kernel buffer → socket buffer (2 copies, no user-space)
        //    Kafka uses Java NIO FileChannel.transferTo() → OS-level zero-copy
        
        // 3. Batching — amortizes network round-trips
        //    Producers batch messages before sending (linger.ms, batch.size)
        //    Consumers fetch in batches (fetch.min.bytes, fetch.max.wait.ms)
        //    Compression applied to entire batch (gzip, snappy, lz4, zstd)
        
        // 4. Page cache — OS-managed, not JVM heap
        //    Kafka stores data on disk but relies on OS page cache for hot data
        //    Avoids JVM GC pressure on large in-memory caches
        //    Consumers often read from page cache (recently produced data)
        
        // ─── Producer configuration — tuning for reliability vs throughput ───
        const producer = new Kafka({ brokers: ['kafka:9092'] }).producer({
          // Reliability settings
          acks: 'all',           // wait for all ISR to acknowledge (strongest durability)
          retries: 5,            // retry on transient failures
          idempotent: true,      // exactly-once within a single producer session (dedup)
        
          // Throughput settings
          batchSize: 16384,      // 16KB batch before sending
          lingerMs: 5,           // wait up to 5ms to accumulate more messages in a batch
          compressionType: 'gzip',
        });
        
        // ─── Consumer configuration ───
        const consumer = kafka.consumer({
          groupId: 'payment-processor',
          sessionTimeout: 30000,     // coordinator marks consumer dead after 30s with no heartbeat
          heartbeatInterval: 3000,   // send heartbeat every 3s
          maxPollIntervalMs: 300000, // max time between poll() calls before rebalance triggered
          autoOffsetReset: 'earliest', // 'earliest' or 'latest' for new groups
        
          // IMPORTANT: disable auto-commit for at-least-once semantics
          autoCommit: false,
        });
        
        await consumer.run({
          eachBatch: async ({ batch, resolveOffset, heartbeat, commitOffsetsIfNecessary }) => {
            for (const message of batch.messages) {
              await processMessage(message);
              resolveOffset(message.offset);
              await heartbeat(); // prevent session timeout during long processing
            }
            await commitOffsetsIfNecessary();
          }
        });

Delivery Semantics

At Most Once

Commit offset before processing. If consumer crashes mid-processing, message is lost. Use when: losing some events is acceptable (metrics, analytics).

At Least Once

Process then commit offset. If crash after processing but before commit, message is redelivered. Consumer must be idempotent. Default for most systems.

Exactly Once

Kafka Transactions: idempotent producer + transactional.id + read-process-write in one atomic transaction. High overhead. Use for financial systems.

Kafka Patterns & Use Cases

// ─── Pattern 1: Event Sourcing with Kafka ───
        // Store all state changes as events in Kafka (compacted topic)
        // Replay the log to rebuild any state at any point in time
        // topic: user-events (compacted, key = userId)
        // Each event: { userId, type: 'USER_CREATED' | 'EMAIL_CHANGED', payload, ts }
        // Consumer rebuilds current state by replaying all events for a userId
        
        // ─── Pattern 2: CDC (Change Data Capture) with Debezium ───
        // Debezium reads PostgreSQL WAL (Write-Ahead Log) and publishes changes to Kafka
        // Topics: postgres.public.users, postgres.public.orders, etc.
        // Each message: { op: 'c'|'u'|'d'|'r', before: {...}, after: {...}, ts_ms }
        // Use case: keep Elasticsearch, Redis, data warehouse in sync with RDBMS
        
        // ─── Pattern 3: CQRS (Command Query Responsibility Segregation) ───
        // Commands go to Kafka → consumers update write model (Postgres)
        //                       → other consumers update read model (Elasticsearch)
        // Queries hit the read model directly (denormalized, fast)
        // Eventual consistency: read model may lag by milliseconds
        
        // ─── Pattern 4: Saga Pattern (distributed transactions) ───
        // Replace 2PC with a sequence of local transactions + compensating events
        // Order service publishes 'ORDER_CREATED'
        //   → Payment service subscribes, charges card, publishes 'PAYMENT_COMPLETED'
        //     → Inventory service reserves stock, publishes 'STOCK_RESERVED'
        //       → Order service marks ORDER_CONFIRMED
        // On failure: compensating events trigger rollback chain
        // Choreography (above) vs Orchestration (Saga orchestrator directs steps)
        
        // ─── Pattern 5: Fan-out ───
        // One producer, multiple consumer groups — each gets full stream independently
        // E.g.: order-events topic consumed by:
        //   - notification-service (sends email/SMS)
        //   - analytics-service (aggregates metrics)
        //   - fraud-detection-service (real-time scoring)
        // No producer changes needed to add new consumers
        
        // ─── Kafka Streams — stream processing on the JVM ───
        // KStream: unbounded stream of records
        // KTable: changelog stream — current value per key (like a materialized view)
        // Join KStream with KTable: enrich events with current state
        // Windowed aggregations: count events per 5-minute tumbling windows
        
        // ─── Node.js: kafkajs ───
        // npm install kafkajs
        const { Kafka, Partitioners } = require('kafkajs');
        const kafka = new Kafka({
          clientId: 'my-app',
          brokers: process.env.KAFKA_BROKERS.split(','),
          ssl: true,
          sasl: { mechanism: 'scram-sha-512', username: process.env.KAFKA_USER, password: process.env.KAFKA_PASS },
        });
        
        // Transactional producer (exactly-once)
        const producer = kafka.producer({ transactionalId: 'order-producer-1', idempotent: true });
        await producer.connect();
        const transaction = await producer.transaction();
        try {
          await transaction.send({ topic: 'orders', messages: [{ key: orderId, value: JSON.stringify(order) }] });
          await transaction.sendOffsets({ consumerGroupId: 'order-consumer', topics: [{ ... }] });
          await transaction.commit();
        } catch (e) {
          await transaction.abort();
        }
Kafka Interview Q&A

Kafka Interview Q&A

What is the difference between Kafka and a traditional message queue (like RabbitMQ or SQS)?
Traditional queues: messages are deleted once consumed. Each message delivered to one consumer. Good for task queues and work distribution. Kafka: messages are persisted to disk for a configurable retention period (days/weeks). Multiple consumer groups can read the same message independently. Consumers control their own offset — can replay from any point. This makes Kafka suitable for event sourcing, CDC, audit logs, and feeding multiple downstream systems from one event stream. RabbitMQ supports complex routing (exchanges, bindings, topics) and is better for routing-heavy use cases. Kafka is better for high-throughput append-only streams, event replay, and fan-out to multiple consumers. SQS is fully managed, simpler, but no replay and per-message billing.
How does Kafka guarantee message ordering? What are the limitations?
Kafka guarantees ordering only within a single partition. Messages with the same key are always routed to the same partition (via key hash), so all events for a given entity (e.g., one user, one order) are ordered. Across partitions, there is no ordering guarantee. This means: if you need strict global ordering across all messages, you must use a single partition (sacrificing throughput). If you need per-entity ordering, use the entity ID as the message key. Limitation: partition rebalance during consumer group changes can cause brief out-of-order delivery if not handled carefully. Idempotent consumers handle duplicate delivery; sequence numbers handle out-of-order delivery within the same partition.
Explain consumer group rebalancing. What causes it and what are the consequences?
A rebalance is triggered when: a new consumer joins the group, an existing consumer leaves or crashes, a new partition is added to the subscribed topic, or maxPollIntervalMs is exceeded (consumer is considered dead). During a rebalance, all consumers stop processing (stop-the-world), partitions are revoked and redistributed. This pause can last seconds to minutes depending on the number of partitions and consumers. Consequences: processing lag increases, at-least-once semantics mean messages may be reprocessed by the new consumer for un-committed offsets. Mitigation: use Cooperative Sticky Assignor (Kafka 2.4+) — only affected partitions are revoked and reassigned, other consumers keep their partitions and continue processing. Tune sessionTimeout and heartbeatInterval to avoid spurious rebalances. Ensure eachMessage/eachBatch processing is fast enough to call poll() within maxPollIntervalMs.
Design a system to process 1 million payment events per second using Kafka.
Partitioning: Create the topic with 100+ partitions (rule of thumb: 1 partition handles ~10K msgs/sec for simple processing). Partition by payment_id or user_id to ensure per-entity ordering. Producer: Use batching (batchSize=512KB, lingerMs=5), compression (lz4 is fastest), acks=all + idempotent=true for durability without duplicates. Consumer: Deploy 100 consumer instances (one per partition). Use eachBatch for bulk processing. Process in parallel within each batch using Promise.all for I/O-bound work. Async commit after batch. Downstream: If writing to Postgres, use COPY or bulk INSERT (not one INSERT per event). Use a write buffer with 100ms flush windows. Dead Letter Queue: any message that fails 3 retries goes to a DLQ topic for manual inspection. Monitoring: Consumer lag per partition (alert if lag > 10K and growing), broker disk usage, ISR shrinking alerts.

HLD

High-Level Design (HLD) Framework

HLD asks you to architect a large-scale distributed system in 45 minutes. You're evaluated on: requirements clarification, capacity estimation, API design, data modeling, component selection, scalability approach, and trade-off reasoning. Use this framework on every system design question.

Step-by-Step Framework

Step 1: Clarify Requirements (5 min)

  • Functional: what must the system do? Core features only (v1)
  • Non-functional: latency target? consistency? availability?
  • Scale: DAU, QPS, data size, read:write ratio
  • Constraints: mobile? global? real-time?

Step 2: Capacity Estimation (3 min)

  • QPS = DAU × actions/day ÷ 86400
  • Storage = records/day × record size × retention days
  • Bandwidth = QPS × avg response size
  • Read/write ratio → drives cache and replica strategy

Step 3–4: API + Data Model (8 min)

  • Define 3–5 core APIs: method, path, request/response
  • Primary entities and relationships
  • SQL vs NoSQL decision with reasoning
  • Key indexes and access patterns

Step 5–7: Design + Deep Dive (25 min)

  • Draw the HLD: clients → CDN → LB → services → DB/cache
  • Let interviewer pick a component to deep dive
  • Discuss bottlenecks and how you'd scale each component
  • Explicitly call out trade-offs: CP vs AP, sync vs async
LLD

Low-Level Design (LLD) & OOP Design

LLD asks you to design the classes, interfaces, and relationships for a specific component — a parking lot, elevator system, chess game, or ATM. Evaluated on: SOLID principles, design patterns, extensibility, and clean OOP.

// ─── LLD: Design a Parking Lot ───
        // Requirements: multiple floors, multiple spot types, entry/exit, billing
        
        // SOLID: each class has one responsibility
        class ParkingSpot {
          constructor(id, floor, type) { // type: COMPACT, REGULAR, LARGE, MOTORCYCLE
            this.id = id; this.floor = floor; this.type = type;
            this.vehicle = null;
          }
          isAvailable() { return this.vehicle === null; }
          park(vehicle) { this.vehicle = vehicle; }
          unpark() { const v = this.vehicle; this.vehicle = null; return v; }
        }
        
        class Vehicle {
          constructor(plate, type) { this.plate = plate; this.type = type; }
        }
        
        // Strategy pattern: pluggable fee calculation
        class HourlyFeeStrategy {
          calculate(hours, type) {
            const rates = { MOTORCYCLE: 1, COMPACT: 2, REGULAR: 3, LARGE: 4 };
            return hours * (rates[type] || 3);
          }
        }
        
        class ParkingTicket {
          constructor(vehicle, spot) {
            this.id = crypto.randomUUID();
            this.vehicle = vehicle;
            this.spot = spot;
            this.entryTime = Date.now();
            this.exitTime = null;
          }
          close() { this.exitTime = Date.now(); }
          durationHours() { return (this.exitTime - this.entryTime) / 3_600_000; }
        }
        
        class ParkingLot {
          constructor(floors, spotsPerFloor) {
            this.floors = floors;
            this.spots = new Map(); // spotId → ParkingSpot
            this.activeTickets = new Map(); // plate → ParkingTicket
            this.feeStrategy = new HourlyFeeStrategy();
            this._initialize(floors, spotsPerFloor);
          }
        
          _initialize(floors, spotsPerFloor) {
            for (let f = 0; f < floors; f++) {
              for (let s = 0; s < spotsPerFloor; s++) {
                const type = s < 2 ? 'MOTORCYCLE' : s < 6 ? 'COMPACT' : s < 9 ? 'REGULAR' : 'LARGE';
                const id = `F${f}S${s}`;
                this.spots.set(id, new ParkingSpot(id, f, type));
              }
            }
          }
        
          // Find nearest available spot for vehicle type
          findSpot(vehicleType) {
            // Order: same floor, nearest spot
            for (const [, spot] of this.spots) {
              if (spot.isAvailable() && this._canFit(vehicleType, spot.type)) return spot;
            }
            return null; // lot full
          }
        
          _canFit(vehicleType, spotType) {
            const sizes = { MOTORCYCLE: 0, COMPACT: 1, REGULAR: 2, LARGE: 3 };
            return sizes[vehicleType] <= sizes[spotType];
          }
        
          park(vehicle) {
            if (this.activeTickets.has(vehicle.plate)) throw new Error('Already parked');
            const spot = this.findSpot(vehicle.type);
            if (!spot) throw new Error('Lot full');
            spot.park(vehicle);
            const ticket = new ParkingTicket(vehicle, spot);
            this.activeTickets.set(vehicle.plate, ticket);
            return ticket;
          }
        
          exit(plate) {
            const ticket = this.activeTickets.get(plate);
            if (!ticket) throw new Error('Not found');
            ticket.close();
            ticket.spot.unpark();
            this.activeTickets.delete(plate);
            const fee = this.feeStrategy.calculate(ticket.durationHours(), ticket.vehicle.type);
            return { ticket, fee };
          }
        
          availableCount() {
            return [...this.spots.values()].filter(s => s.isAvailable()).length;
          }
        }
        
        // Key design patterns used:
        // Strategy: HourlyFeeStrategy — swap in WeekendFeeStrategy without changing ParkingLot
        // Observer: ParkingLot could emit events for capacity alerts
        // Singleton: ParkingLot instance managed by IoC container
        // Factory: VehicleFactory to create correct vehicle type from input

SOLID Principles — Rapid Reference

S

Single Responsibility

One class = one reason to change

O

Open/Closed

Open for extension, closed for modification

L

Liskov Substitution

Subtypes must be substitutable for base types

I

Interface Segregation

Clients shouldn't depend on unused methods

D

Dependency Inversion

Depend on abstractions, not concretions

Design WhatsApp / Messaging System

// ─── Scale: 2B users, 100B msgs/day ───
        // QPS: 100B / 86400 ≈ 1.2M msgs/sec
        // Storage: avg 100 bytes × 100B = 10TB/day
        
        // ─── Core architecture decisions ───
        
        // 1. Transport: WebSocket for persistent bidirectional connection
        //    Long polling fallback for mobile networks
        //    Connection server per user: user connects to specific chat server
        //    Service discovery (ZooKeeper) maps userId → chatServerId
        
        // 2. Message routing flow:
        //    Sender → sender's chat server
        //    → check if recipient online (Redis presence)
        //    → YES: send to recipient's chat server → push via WebSocket
        //    → NO: store in offline message queue (Redis or SQS) + send push notification
        
        // 3. Message storage: Apache Cassandra
        //    Why Cassandra: wide-column, optimized for time-series append
        //    Schema: partition_key=(chat_id), clustering_key=(message_id DESC)
        //    → O(1) write, O(1) range read for last N messages per chat
        //    Snowflake IDs for message_id: timestamp bits (41) + machine (10) + sequence (12)
        //    → global uniqueness, sortable by time, no coordination needed
        
        // 4. Media (images, video): pre-signed S3 URLs
        //    Client uploads directly to S3 (bypasses chat servers)
        //    Only metadata (S3 URL) sent via chat servers
        //    CDN in front of S3 for delivery
        
        // 5. Group messaging:
        //    Up to 256 members: fan-out on write (message copied to each member's mailbox)
        //    Above 256 members: fan-out on read (store once, each client reads from shared partition)
        
        // 6. End-to-end encryption (WhatsApp Signal protocol):
        //    Keys exchanged out-of-band (server never holds private keys)
        //    Server stores only encrypted ciphertext
        //    Read receipts: double tick (delivered), blue tick (read) — ACK sent from client
        
        // 7. Presence service (online/last seen):
        //    Redis SET userId → lastHeartbeatTs with TTL 60s
        //    Client sends heartbeat every 30s
        //    GET userId → if missing from Redis = offline; compute "last seen" from last heartbeat

Design Uber / Ride-Sharing

// ─── Scale: 5M rides/day, 3M active drivers ───
        // QPS: ~58 requests/sec for bookings; driver location updates: 3M × (1/5 sec) = 600K/sec
        
        // ─── Core challenges ───
        // 1. Geospatial matching: find nearby available drivers in O(log n)
        // 2. Surge pricing: calculate in real-time based on supply/demand
        // 3. Driver location: handle 600K location updates/second
        // 4. Trip state machine: atomically transition states across services
        
        // ─── Architecture ───
        
        // Driver location service:
        // Geohash: encode lat/lng as a base32 string — adjacent cells share prefix
        // Resolution 6 geohash ≈ 1.2km × 0.6km precision
        // Redis GEOADD: stores (lng, lat) in a sorted set using geohash scoring
        //   GEOADD drivers:available lng lat "driver-123"
        //   GEORADIUS drivers:available lng lat 3 km WITHCOORD WITHDIST COUNT 10 ASC
        // Driver publishes location every 5 sec via WebSocket to location service
        // Location service writes to Redis (TTL 30s — driver disappears if no update)
        
        // Matching service:
        // When user requests ride: GEORADIUS to find nearest 10 drivers
        // Sort by ETA (call routing service or use Haversine distance as proxy)
        // Dispatch to nearest driver via push notification
        // Driver has 15s to accept; if declined, move to next candidate
        // Optimistic: pre-compute driver ETAs using road network (OSRM or Google Maps API)
        
        // Trip service — state machine:
        // States: REQUESTED → DRIVER_ASSIGNED → DRIVER_ARRIVING → IN_PROGRESS → COMPLETED
        // Each transition is an event published to Kafka
        // State stored in Postgres (ACID — financial accuracy matters)
        // Kafka consumers: billing, analytics, support, notifications
        
        // Surge pricing:
        // Supply/demand ratio per geohash × time window
        // demand_ratio = pending_requests / available_drivers
        // if ratio > 1.5 → surge = 1.2x; > 2.0 → 1.5x; > 3.0 → 2.0x
        // Computed by a streaming job (Flink/Spark Streaming) on rider request events
        // Cached in Redis with 1-minute TTL
        
        // Payments:
        // Pre-authorize card at ride request
        // Capture actual amount at trip completion
        // Fare = base_fare + (per_mile × distance) + (per_min × duration) × surge_multiplier
        // Stripe or Braintree for payment processing; idempotency keys on all payment calls
        
        // Data storage choices:
        // Driver locations: Redis (ephemeral, sub-ms reads)
        // Trips: PostgreSQL (ACID, relational, financial)
        // Ride history (analytics): S3 + Redshift (columnar, cheap long-term)
        // Real-time geospatial: Redis GEOSEARCH + PostGIS for complex polygons (surge zones)

Design Netflix / Video Streaming

// ─── Scale: 200M subscribers, 15% of global internet traffic ───
        // 2 billion hours streamed per month ≈ 1.4M concurrent streams
        
        // ─── Upload pipeline (content ingestion) ───
        // 1. Studio uploads raw video file (up to 4K, terabytes)
        // 2. Transcoding service converts to multiple formats:
        //    - Resolutions: 240p, 360p, 480p, 720p, 1080p, 4K
        //    - Codecs: H.264 (compatibility), H.265/HEVC (efficiency), AV1 (newest)
        //    - Audio: multiple languages, audio descriptions
        // 3. Each video → ~1200 encoding variants per title
        // 4. Chunked into 2-4 second segments (MPEG-DASH / HLS manifests)
        // 5. Stored on S3; metadata in Cassandra; manifests in S3
        
        // ─── Adaptive Bitrate Streaming (ABR) ───
        // Client downloads manifest (.m3u8 or .mpd) listing all quality levels
        // Player monitors bandwidth and buffer health in real-time
        // ABR algorithm picks quality level per chunk:
        //   - Buffer-based: if buffer > 30s → use highest quality
        //   - Bandwidth-based: estimate bandwidth from last download speed
        //   - Hybrid (Netflix BOLA): minimizes stalls, maximizes quality
        // Segment URL: cdn.example.com/{videoId}/1080p/chunk_0042.ts
        
        // ─── CDN strategy ───
        // Netflix Open Connect: Netflix's own CDN (ISP-embedded servers)
        //   Netflix ships 100TB+ appliances to major ISPs for free
        //   Popular titles cached at ISP level — last-mile latency eliminated
        // Cache hierarchy: browser → ISP CDN → regional CDN → origin S3
        // Pre-positioning: top 1000 titles pushed to CDN during off-peak hours (nightly)
        
        // ─── Recommendation system ───
        // Offline: collaborative filtering on viewing history (Apache Spark ML)
        // Near-real-time: Kafka → feature store → model serving (TensorFlow Serving)
        // Personalized thumbnails: A/B tested per user — same title, different image
        // Row ordering: each row (Continue Watching, Trending, etc.) scored per user
        
        // ─── Resilience: Netflix Chaos Engineering ───
        // Hystrix: circuit breaker library — fallback to degraded mode on service failure
        // Chaos Monkey: randomly kills production instances — forces fault-tolerant design
        // Simian Army: chaos testing across region failures, latency injection, etc.
        // Fallback strategy: if recommendation service fails → serve curated editorial list
        
        // ─── Database choices ───
        // User data (accounts, billing): MySQL (strong consistency)
        // Viewing history: Cassandra (time-series, append-only, globally distributed)
        // Metadata (title info, cast): Elasticsearch (full-text search) + DynamoDB (primary)
        // A/B testing config: Archaius (dynamic property service, ZooKeeper-backed)
        // Analytics: Apache Iceberg on S3 + Spark (petabyte-scale offline analytics)

Design Twitter / Social Feed

// ─── Scale: 350M users, 500M tweets/day, 150M DAU ───
        // Home timeline: each user follows avg 200, each has avg 100 followers
        // QPS: 500M / 86400 ≈ 5800 writes/sec; timeline reads ≈ 300K QPS
        
        // ─── The core challenge: fan-out on write vs fan-out on read ───
        
        // OPTION A: Fan-out on Write (push model)
        // When user tweets → immediately copy tweet_id to each follower's timeline cache
        // Read home timeline: O(1) — just read pre-built cache
        // Write tweet: O(n) where n = followers count (50 for average user, 100M for celebrities)
        // Twitter uses this for users with < ~100K followers
        // Storage: Redis sorted set per user: ZADD timeline:userId score tweetId
        //          score = tweet timestamp (for chronological ordering)
        
        // OPTION B: Fan-out on Read (pull model)
        // When user reads timeline → fetch tweets from all followed users, merge, sort
        // Read: O(n) where n = follows count — expensive for users following 10K accounts
        // Write: O(1) — just store the tweet
        // Twitter uses this for celebrity accounts (Lady Gaga, Barack Obama)
        
        // HYBRID: Twitter's actual approach
        // Regular users (<= ~100K followers): fan-out on write → all followers' caches updated
        // Celebrities (> threshold): excluded from fan-out; pulled and merged at read time
        // Client sees: pre-built timeline cache + merge of celebrity tweets on the fly
        
        // ─── Tweet storage ───
        // MySQL: tweet metadata (id, userId, text, mediaUrls, createdAt, likes, retweets)
        // Tweet ID: Snowflake — 64-bit: timestamp(41) + datacenter(5) + machine(5) + sequence(12)
        // Media: S3 + CDN (images, videos transcoded like Netflix)
        
        // ─── Timeline service ───
        // Redis cluster: sorted set per user, keyed by timestamp
        //   ZADD home:timeline:userId timestamp tweetId
        //   ZREVRANGE home:timeline:userId 0 19 → last 20 tweet IDs
        // Then: batch-fetch tweet content: MGET tweet:id1 tweet:id2 ...
        // Limit timeline cache to 800 most recent tweets (cold storage = MySQL)
        
        // ─── Search ───
        // Real-time tweet indexing via Earlybird (Lucene-based, custom inverted index)
        // Optimized for recency: index segments rotated hourly, most recent searched first
        // Trending topics: sliding-window count of hashtags over 24h, ranked by velocity
        
        // ─── Notification service ───
        // Push: user tweets, followers with notifications on receive APNs/FCM push
        // In-app: Kafka event → notification worker → write to notification table → SSE/WebSocket
        // Unread counts: Redis INCR notif:count:userId on event; reset on page load
        
        // ─── Retweet / Like ───
        // Like: INCR tweet:likes:tweetId in Redis + async write to MySQL
        // Retweet: creates new tweet record with retweetOf = originalTweetId
        //          triggers fan-out as normal tweet for retweeter's followers
        
        // ─── Rate limiting ───
        // API: 1500 tweet reads per 15 min per user (sliding window in Redis)
        // Token bucket per userId: HINCRBY ratelimit:userId window_key 1
        // Check + decrement atomic via Lua script
Interview Q&A

HLD / LLD Interview Q&A

How do you choose between SQL and NoSQL for a new system?
Choose SQL (PostgreSQL/MySQL) when: you have complex relationships requiring JOINs, you need ACID transactions (financial, booking systems), your schema is stable, you need rich querying (aggregations, range queries), team is familiar with relational modeling. Choose NoSQL when: schema is flexible/evolving rapidly (document store), you need horizontal write scaling beyond what a single Postgres primary can handle (Cassandra), your access patterns are narrow and predictable (key-value: Redis, DynamoDB), you need geospatial queries (MongoDB), you need time-series at massive scale (InfluxDB, TimescaleDB). Most modern systems use both: Postgres for transactional data, Redis for caching, Elasticsearch for search, Cassandra for time-series. The question isn't which — it's right tool for each access pattern.
Explain the CAP theorem with a real example.
CAP theorem states that a distributed system can guarantee at most two of: Consistency (all nodes see the same data at the same time), Availability (every request gets a response), Partition Tolerance (system continues operating despite network partitions). Network partitions are inevitable in distributed systems — so the real choice is between CP and AP during a partition. CP example (bank transfer): if two data centers can't communicate, block all writes until partition resolves. Stale data for a financial transaction is unacceptable. Systems: HBase, ZooKeeper, MongoDB with majority write concern. AP example (shopping cart): if data centers can't communicate, let users continue adding to cart. Reconcile conflicts when partition heals (CRDT/last-write-wins). Systems: Cassandra (eventual consistency), DynamoDB (tunable consistency), CouchDB. PACELC extends CAP: even when there's no partition, there's a latency/consistency trade-off. Cassandra optimizes for low latency (more replicas → slower writes to achieve consistency).
Design a URL shortener like bit.ly. Walk through the key design decisions.
Requirements: shorten URL, redirect, analytics. Scale: 100M new URLs/day, 10B redirects/day. Storage: 100M × 500 bytes = 50GB/day. Redirect QPS: 10B / 86400 ≈ 115K/sec. Short code generation: base62 (a-z, A-Z, 0-9). 7 characters = 62^7 = 3.5 trillion unique URLs. Options: (1) MD5 hash of long URL, take first 7 chars — risk of collision, check DB. (2) Auto-increment ID from DB, encode in base62 — deterministic, no collision, but ID sequence is guessable. (3) Pre-generate short codes using a Key Generation Service (KGS) that creates random codes in bulk and stores them in a "keys" table — checked out atomically to avoid races. Database: writes to MySQL (100M/day is manageable). Reads: heavily cached in Redis (hot URLs). Redirect flow: client hits bit.ly/abc123 → CDN edge → check Redis cache → if miss, query MySQL → 301 (permanent, browser caches) or 302 (temporary, analytics tracked). Analytics: Kafka event per redirect → stream processor → aggregated stats in ClickHouse. Custom URLs: allow users to specify slug, unique constraint on short_code column.
Design a rate limiter. What algorithms exist and where would you deploy it?
Algorithms: (1) Token Bucket — bucket fills at rate r, each request consumes a token. Allows bursts up to bucket capacity. Implementation: (tokens, lastRefill) per user in Redis, compute available tokens on each request. (2) Sliding Window Counter — track request count in the current window using a weighted blend of previous and current fixed window counts. More accurate than fixed window. (3) Leaky Bucket — requests enter a queue (bucket) and are processed at a fixed rate. Smooths bursts but adds latency. (4) Fixed Window — simplest, but vulnerable to 2× spike at window boundary. Deployment layers: (A) API Gateway level (Kong, AWS API Gateway) — centralized, applies before requests reach services, cheapest to operate. (B) Application level — each service enforces its own limits, enables per-endpoint limits, but requires Redis for distributed state. (C) Client-side — request throttling in SDK, rate limit feedback via 429 + Retry-After header. Redis implementation: INCR ratelimit:{userId}:{window} EXPIRE ratelimit:{userId}:{window} 60 — atomic increment and set expiry. For sliding window: Redis sorted set with timestamp as score, count members in last 60 seconds.
What are the differences between horizontal and vertical scaling? When does each break down?
Vertical scaling (scale up): add more CPU/RAM/disk to existing server. Simple — no code changes, no distributed coordination. Breaks down: hardware limits (largest EC2 instance has finite CPU/RAM), cost grows superlinearly at high tiers, single point of failure, zero-downtime upgrades are hard. Horizontal scaling (scale out): add more instances, distribute load. Enables near-linear throughput growth. Requires: stateless services (session state in Redis, not in-process), load balancer, distributed data access, coordination for distributed tasks. Breaks down: state synchronization complexity, network latency between nodes, distributed transactions are expensive, operational complexity grows with fleet size. In practice: vertical scale until you hit diminishing returns or >$5K/month on a single instance, then horizontal scale. Databases typically go vertical first (Postgres handles massive load on a single large instance), then read replicas, then sharding — each step adds massive complexity. Stateless application servers should be horizontal from the start.
What is the difference between synchronous and asynchronous communication between microservices? When is each appropriate?
Synchronous (REST/gRPC): caller waits for response. Request-response contract, immediate consistency, simpler to reason about, natural error propagation. Downsides: tight coupling — if downstream is slow/down, upstream blocks and times out. Cascading failures: one slow service can saturate thread pools and bring down callers. Use for: read operations that need immediate results, user-facing operations requiring real-time response, simple request-response workflows. Asynchronous (Kafka, SQS, RabbitMQ): caller publishes event and continues. Downstream processes independently. Benefits: loose coupling — services can deploy independently, producer doesn't care if consumer is slow. Resilient to downstream failures — messages queue up. Natural load leveling — consumer processes at its own pace. Downsides: eventual consistency — caller doesn't know when processing completes, harder to debug (distributed tracing needed), harder to handle errors (DLQ, idempotency). Use for: long-running operations, notifications, fan-out to multiple consumers, write amplification (fanout on write), audit logs, cross-domain events. Guideline: if the user needs an immediate answer — synchronous. If you're notifying other systems of something that happened — asynchronous.

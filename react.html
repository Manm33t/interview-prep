React Fundamentals

Understanding what React actually does under the hood — not just how to use it — separates senior engineers from mid-level ones. Interviewers probe JSX compilation, the Virtual DOM diffing algorithm, and subtle component design decisions constantly.

JSX Internals — What Babel Actually Produces

JSX is syntactic sugar. Babel transforms every JSX element into a React.createElement(type, props, ...children) call. React 17+ uses the new JSX transform which imports from react/jsx-runtime automatically. The result is a plain JavaScript object — a React element.

// React.createElement output:
        // { $$typeof: Symbol(react.element), type: 'div', key: null, ref: null,
        //   props: { className: 'card', children: [...] } }
        // $$typeof: Symbol prevents XSS — Symbols can't be serialized to JSON
        
        // React 17+ new JSX transform (no React import needed):
        import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime';
        const element = _jsxs('div', { className: 'card', children: [ _jsx('h1', { children: 'Hello' }) ] });
        
        // Functional vs Class — why functional won:
        // 1. No `this` binding confusion
        // 2. Easier composition via custom hooks vs HOCs/render props
        // 3. React concurrent features designed around functional model
        
        // Keys — stable identity in lists
        // BAD: key={index} — reorders break component state
        // GOOD: key={item.id} — stable, data-driven identity
        // KEY TIP: changing key = force remount (resets all state)
        function ParentWithReset() {
          const [key, setKey] = useState(0);
          return (
            <>
              <ExpensiveForm key={key} />
              <button onClick={() => setKey(k => k + 1)}>Reset Form</button>
            </>
          );
        }
        
        // Reconciliation: O(n) heuristic diffing
        // Rule 1: Different type = full remount (state is lost)
        // Rule 2: Same type = update props only
        // Fiber: interruptible linked-list traversal (React 16+)
        // Phase 1 (render): interruptible, no side effects
        // Phase 2 (commit): synchronous, applies DOM mutations
Virtual DOM misconception: It's fast because it batches and minimizes DOM mutations, not because DOM is slow. React's commit phase applies all mutations in one pass, avoiding layout thrashing.

Controlled vs Uncontrolled Components

// Controlled: React owns state — every keystroke → setState → instant validation
        function ControlledForm() {
          const [email, setEmail] = useState('');
          return (
            <input type="email" value={email}
              onChange={e => setEmail(e.target.value)} />
          );
        }
        
        // Uncontrolled: DOM owns state, ref reads on demand
        function UncontrolledForm() {
          const emailRef = useRef(null);
          const handleSubmit = (e) => {
            e.preventDefault();
            console.log(emailRef.current.value);
          };
          return <form onSubmit={handleSubmit}><input ref={emailRef} /></form>;
        }
        // Uncontrolled preferred for: file inputs, non-React library integration, perf on large forms

React Hooks — Deep Dive

Hooks are index-keyed linked lists attached to the fiber, which is why the rules of hooks exist. Senior engineers must understand the mental model, not just the API.

useState — Batching, Functional Updates, Lazy Initializer

// Functional update — always use when new state depends on old state
        const incrementThrice = () => {
          setCount(c => c + 1);
          setCount(c => c + 1);
          setCount(c => c + 1); // correctly goes 0 → 3
        };
        
        // Lazy initializer — function runs ONCE on mount
        const [data, setData] = useState(() => parseHugeJSON(rawInput)); // not called every render
        
        // React 18 automatic batching — ONE re-render for all updates
        const handleAsync = async () => {
          await fetchSomething();
          setA(1);  // batched!
          setB(2);  // batched! → single re-render
        };
        // React 17: setTimeout / Promise updates NOT batched — two renders
        // Opt out: import { flushSync } from 'react-dom'; flushSync(() => setA(1));

useEffect — Lifecycle, Cleanup, Common Pitfalls

// Race condition fix — cleanup flag
        function DataFetcher({ userId }) {
          const [data, setData] = useState(null);
          useEffect(() => {
            let cancelled = false;
            async function fetchUser() {
              const res = await fetch(`/api/users/${userId}`);
              const json = await res.json();
              if (!cancelled) setData(json); // guard against stale update after unmount
            }
            fetchUser();
            return () => { cancelled = true; }; // cleanup
          }, [userId]);
        }
        
        // Mistake: Object/function in deps → infinite loop
        // BAD: const options = { filter }; useEffect(() => {}, [options]); // new ref every render
        // FIX: useEffect(() => fetchData({ filter }), [filter]); // stable primitive
        
        // Mistake: Missing cleanup for event listeners
        useEffect(() => {
          const handler = (e) => setPos({ x: e.clientX, y: e.clientY });
          window.addEventListener('mousemove', handler);
          return () => window.removeEventListener('mousemove', handler); // MUST cleanup
        }, []);
        
        // React 18 Strict Mode: mounts → unmounts → remounts in dev to catch bad cleanup

useRef, useContext, useReducer, useCallback, useMemo

// useRef: mutable value that does NOT trigger re-render
        // Use for: DOM refs, timer IDs, previous value tracking, stale closure fix
        const countRef = useRef(count);
        countRef.current = count; // always up-to-date in long-lived callbacks
        
        // forwardRef + useImperativeHandle: expose specific imperative API
        const Modal = React.forwardRef(function Modal(props, ref) {
          const [open, setOpen] = useState(false);
          React.useImperativeHandle(ref, () => ({ open: () => setOpen(true), close: () => setOpen(false) }));
          return open ? <div className="modal">{props.children}</div> : null;
        });
        
        // useContext: ALWAYS memoize the value object!
        const value = useMemo(() => ({ theme, toggle }), [theme]); // else re-renders all consumers
        
        // Split contexts by update frequency — read context vs dispatch context
        const ThemeDispatchContext = React.createContext(null); // stable — write operations
        const ThemeStateContext    = React.createContext(null); // changes — read operations
        
        // useReducer wins over useState when: multiple related fields, named transitions, testable logic
        function fetchReducer(state, action) {
          switch (action.type) {
            case 'FETCH_START':   return { ...state, loading: true, error: null };
            case 'FETCH_SUCCESS': return { loading: false, data: action.payload, error: null };
            case 'FETCH_ERROR':   return { loading: false, data: null, error: action.payload };
            default: throw new Error(`Unknown: ${action.type}`);
          }
        }
        
        // useMemo / useCallback — profile first, memoize second
        // useCallback: stable function reference for React.memo'd child components
        const handleSave = useCallback(() => saveToServer(name), [name]);
        // useMemo: expensive computation
        const filtered = useMemo(() => products.filter(p => p.name.includes(q)), [products, q]);

Custom Hooks — Rules and Common Patterns

// Rules of Hooks: why call order matters
        // React uses CALL ORDER to match hooks to fiber slots (no name-based lookup)
        // Conditional call = slot shift = wrong state → "Rendered fewer hooks than expected"
        
        // useFetch — abort signal handles unmount cleanup
        function useFetch(url) {
          const [state, dispatch] = useReducer(fetchReducer, { status: 'idle', data: null, error: null });
          useEffect(() => {
            if (!url) return;
            const controller = new AbortController();
            dispatch({ type: 'FETCH_START' });
            fetch(url, { signal: controller.signal })
              .then(r => r.json())
              .then(data => dispatch({ type: 'FETCH_SUCCESS', payload: data }))
              .catch(err => { if (err.name !== 'AbortError') dispatch({ type: 'FETCH_ERROR', payload: err.message }); });
            return () => controller.abort();
          }, [url]);
          return state;
        }
        
        // useDebounce — delays value until typing stops
        function useDebounce(value, delay = 300) {
          const [debounced, setDebounced] = useState(value);
          useEffect(() => {
            const timer = setTimeout(() => setDebounced(value), delay);
            return () => clearTimeout(timer);
          }, [value, delay]);
          return debounced;
        }

Profile first with React DevTools Profiler, then apply targeted fixes. Blindly adding React.memo and useMemo everywhere is an anti-pattern.

React.memo — Shallow Comparison

// Skips re-render if props shallowly equal
        const ExpensiveChart = React.memo(function Chart({ data, title }) {
          return <canvas>...</canvas>;
        });
        
        // Objects/arrays: NEW reference = always re-renders
        // Fix: stable reference with useMemo
        const chartData = useMemo(() => computeChartData(userId), [userId]);
        
        // Custom comparison (use sparingly):
        const Chart = React.memo(Chart, (prev, next) => prev.data.id === next.data.id);
        // true = skip re-render; false = re-render

const AccordionContext = React.createContext(null);
        
        function Accordion({ children, defaultOpen = null }) {
          const [openItem, setOpenItem] = useState(defaultOpen);

HOC Pattern

function withAuth(WrappedComponent) {
          function AuthGuard(props) {
            const { user, loading } = useAuth();

Error Boundaries + Portals

// Error boundaries MUST be class components
        class ErrorBoundary extends React.Component {
          state = { hasError: false };
          static getDerivedStateFromError() { return { hasError: true }; }
          componentDidCatch(error, info) { errorTracker.capture(error, info); }
          render() {
            return this.state.hasError
              ? this.props.fallback
              : this.props.children;
          }
        }

State Management — Picking the Right Tool

Decision framework: One component? → useState. Nearby components? → lift state. Many/deep components? → Context or Zustand. Data from server? → React Query. Large app, complex client state, time-travel debugging? → Redux Toolkit.

Zustand — Simple Global State

import { create } from 'zustand';
        import { persist } from 'zustand/middleware';
        
        const useCartStore = create(persist(
          (set, get) => ({
            items: [],
            addItem: (item) => set(state => ({ items: [...state.items, item] })),
            clearCart: () => set({ items: [] }),
            get totalItems() { return get().items.length; },
          }),
          { name: 'cart-storage' } // localStorage persistence
        ));
        
        // Selective subscription — only re-renders when slice changes
        const total = useCartStore(s => s.totalItems);
        const addItem = useCartStore(s => s.addItem); // stable — no re-render

Interview Prep

React Interview — Deep Q&A

Explain React reconciliation and what the Fiber architecture adds.
Reconciliation determines what changed in the UI and applies minimal DOM mutations. React maintains two trees: current (on screen) and work-in-progress (being built). The original stack reconciler (React 15) was recursive and synchronous — could not be interrupted, causing dropped frames on large trees. Fiber (React 16+) re-implements this as an iterative linked-list traversal. Each fiber is a unit of work. React can pause, resume, abort, or reuse work. Two phases: (1) Render phase (interruptible) — traverse fiber tree, mark effects; (2) Commit phase (synchronous) — apply DOM mutations. This enables concurrent features: useTransition marks updates as interruptible so user-facing updates (typing) preempt expensive background renders.
What are the rules of hooks and why do they exist?
Two rules: (1) only call hooks at the top level — never inside conditionals, loops, or nested functions; (2) only call from React function components or custom hooks. React tracks hook state by call order — each component fiber has a linked list of "hook slots." First useState call gets slot 0, second gets slot 1, etc. If a hook is skipped in one render, every hook after shifts down, reading from the wrong slot — silent corruption. Outside rendering, the React dispatcher is cleared, so useState throws "Invalid hook call." ESLint's react-hooks/rules-of-hooks catches these statically.
What is a stale closure in React hooks and how do you fix it?
A stale closure occurs when a function captures a variable at render time, but the variable changes in subsequent renders while the function references the old value. Classic example: setInterval with empty deps captures initial count=0, always sets to 1. Fix 1: functional update — setCount(c => c + 1) doesn't need to close over count. Fix 2: add to deps (re-creates interval on each change). Fix 3: useRef — countRef.current = count always up-to-date in the long-lived callback without restarting the effect.
React 18 automatic batching — how does it differ from React 17?
React 17: updates batched only inside React event handlers (onClick, etc.). setTimeout, Promise.then, native DOM event listeners caused separate re-renders per setState. React 18: automatic batching everywhere — all setA(1); setB(2); calls within one event tick cause ONE re-render, regardless of where they're called (setTimeout, Promise, async function). Opt out with flushSync() when you genuinely need intermediate DOM reads between updates. Requires createRoot() instead of ReactDOM.render to enable.
How does React Query differ from Redux for server state?
Mental model difference: Redux treats server data as application state you manually synchronize. React Query treats it as a cache of remote state that's always potentially stale and needs automatic revalidation. React Query provides: caching, deduplication, background refresh, stale-while-revalidate, optimistic updates, retry with backoff, devtools — all for free. Redux for server state requires ~80 lines per endpoint of actions/thunks/reducer/loading/error. Best practice: use both — React Query for server state, Zustand/Redux for client-only state (undo/redo, local workflow, cross-entity derived state).
What causes hydration mismatches in SSR and how do you fix them?
SSR renders HTML on server; hydration attaches React to existing DOM client-side. Mismatch occurs when server HTML ≠ client render output. Common causes: (1) Date/time — server renders at request time, client renders later → different strings; (2) browser APIs (window, localStorage) called during render — don't exist on server; (3) invalid HTML nesting — browser auto-corrects, changing tree shape. Fixes: use useEffect to populate time-sensitive values client-side; typeof window guard for browser APIs; suppressHydrationWarning for intentional differences. React 18 logs exact mismatched component and value in dev.
useEffect vs useLayoutEffect — when do you use each?
useEffect fires asynchronously after browser paint — non-blocking, default choice. useLayoutEffect fires synchronously after DOM mutations but before browser paints — blocks paint. Use useLayoutEffect only when reading DOM layout and synchronously re-rendering to avoid visual flash: tooltip positioning (read DOM rect, set position, user never sees wrong position), third-party DOM library initialization. IMPORTANT: useLayoutEffect doesn't run on the server (SSR) — use useIsomorphicLayoutEffect = typeof window !== 'undefined' ? useLayoutEffect : useEffect.
Explain the trade-offs between prop drilling, Context, and external state managers.
Prop drilling: simple, explicit data flow, breaks down at 3+ levels — every intermediate component must be modified when the prop shape changes. Context: any component can read without threading props, breaks down when: (a) value changes frequently → all consumers re-render; (b) context grows into a kitchen-sink blob. Fix: split into multiple small focused contexts by update frequency. External managers (Zustand, Redux): solve context's performance problem via selective subscriptions — components subscribe to only the slice they need. Breaks down when overused for server state.
How would you diagnose and fix dropped frames in a React app?
Step 1: Profile — React DevTools Profiler, record the interaction, check Flame Graph for components >16ms. "Why did this render?" shows changed props/state/context. Step 2: Enable "Highlight Updates" — unexpected flashing shows unnecessary re-renders. Step 3: Targeted fixes: large list → react-window virtualization; expensive computation → useMemo; function reference instability → useCallback + React.memo; context thrashing → split contexts or Zustand selectors; main thread blocking → Web Worker; non-urgent updates → useTransition. Step 4: Measure again — verify the fix moved the metric.
When would you use keys outside of lists?
Keys force remounting — changing a key unmounts the old component instance and mounts a fresh one, resetting all internal state. Use cases: (1) Reset a form when the entity changes — <UserForm key={userId} />: changing userId forces full remount, all useState resets; (2) Force re-animation on data change; (3) Reset error boundary state — <ErrorBoundary key={resetKey}>: incrementing resetKey resets the boundary. This is the canonical React pattern, preferred over componentWillReceiveProps comparisons or imperative reset methods.
